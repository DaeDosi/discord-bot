# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Architecture

This is a monorepo for **NexBot**, a Discord bot with a web dashboard, split into independently-run services:

- **Root (`main.py`, `cogs/`, `database/`, `utils/`)** — the Discord bot itself (discord.py). Cogs are loaded from the `COGS` list in `main.py`; each new cog must be added there to load.
- **`web/backend/`** — FastAPI dashboard API (separate process, separate `requirements.txt`, its own `Dockerfile`). Routers live in `web/backend/routers/`. Discord OAuth login issues a JWT (`web/backend/auth.py`); CORS allows `*` because auth uses the `Authorization` header, not cookies.
- **`web/frontend/`** — Next.js dashboard (`app/dashboard/[guildId]/...` per-feature settings pages). The guild-scoped layout (`app/dashboard/[guildId]/layout.tsx` + `Navbar`) is full-width (no `max-w-*` cap) by design — don't reintroduce a centered narrow column there. `components/Sidebar.tsx` self-fetches `api.guilds.list()` to render a guild switcher (icon/name button that opens a dropdown of the user's other managed guilds); it no longer takes a `guildName` prop.
- **`relay/relay.py`** — standalone long-running process meant to run on a Korea-IP host (e.g. Oracle Cloud VM), separate from Railway. It polls Chzzk's unofficial community API (which geo-blocks non-Korean IPs) and forwards new posts to the Railway backend via a webhook secured with `RELAY_SECRET`.
- **`chzzk-repo/`** — separate Expo/React Native app (`chzzk-noti-app`), early-stage, unrelated to the web dashboard's build.

**Chzzk has two independent linkage flows — don't conflate them:**
1. **Streamer OAuth** (`web/backend/routers/chzzk_auth_router.py`, dashboard "방송설정 → 치지직" page): the streamer links their own Chzzk account so the bot can read their follower list and (as of the realtime chat feature) subscribe to/post in their live chat. Tokens land in `chzzk_subscriptions.streamer_access_token`/`streamer_refresh_token`.
2. **Viewer verification** (`web/backend/routers/verify_router.py` + `chzzk_auth_router.py`'s viewer flow, dashboard "입장 인증" page): an individual Discord member links *their own* Chzzk account so the bot can map `discord user_id ↔ chzzk_channel_id`. Rows land in `chzzk_verifications`.

Chat-triggered features (출석체크 등, `cogs/chzzk_chat.py`) only work for viewers who did flow 2 — a streamer who only did flow 1 is not "known" to the checkin logic, even in their own chat.

- **`cogs/chzzk_chat.py`** — runs in the bot process (not web/backend). Subscribes to a streamer's live chat via Chzzk's *official* Session API (not chat scraping) using the [`chzzkpy`](https://pypi.org/project/chzzkpy/) library, reusing the streamer's OAuth token from flow 1. Library-internals gotchas worth knowing before touching this file again:
  - `Client.connect()`'s return value is the Engine.IO transport id, **not** the `sessionKey` the subscribe REST call needs — the real sessionKey only arrives via the `on_connect` event payload.
  - `chzzkpy` doesn't surface a "the socket died" callback (a failed heartbeat just silently kills a background task), so liveness is checked by peeking at `Client._gateway[transport_sid].is_connected` each sync cycle (`SYNC_INTERVAL_SECONDS`, currently 60s) and forcing a full reconnect if it's gone.
  - `chzzkpy`'s `subscribe()` throws `HTTPException` with a Korean "이미 구독 중인 이벤트입니다" (409) message when the platform-side subscription is still alive from before a reconnect — this is treated as success (string-matched), not an error, or the channel would never make it back into `self._channels`.
  - `chzzkpy` has no `is_broadcaster`/`is_streamer` field on `Message` — the whole codebase's "is this the streamer chatting in their own channel" check is the heuristic `message.user_id == message.channel` (chzzk's chat channel id and the broadcaster's own channel id are the same value). There's no more reliable alternative currently available.
  - `chzzkpy`'s `Message` model can be constructed directly (bypassing the real socket) with alias kwargs (`senderChannelId=`, `channelId=`, `chatChannelId=`, `messageTime=`, `profile={...}`) — this is how the dashboard's web test-message feature (below) simulates chat without a live connection.
  - Chat commands are configured per guild in `chzzk_chat_commands` (`checkin` type, max 1 per guild; `reply` type, max 5) and enforced once-per-day via `chzzk_checkin_log`. Trigger text **cannot contain whitespace** (the parser splits `!trigger arg` on the first space to support arguments like `!투표 1`, so a multi-word trigger could never match) and cannot collide with the reserved built-in command names below — both are validated in `chzzk_router.py`'s `_validate_trigger`. A rolling debug feed of raw in/out chat lines is kept in `chzzk_chat_log` (last 50 per guild) and surfaced in the dashboard's "실시간 채팅 명령어" tab for exactly this kind of "is it actually receiving anything" debugging.
  - `chzzk_subscriptions.chat_enabled` is the single master switch for the whole realtime-chat feature (`_sync_channels` filters on it directly in SQL) — it does **not** depend on whether any `chzzk_chat_commands` rows exist, so a guild with zero configured commands still connects and gets the built-in commands below. Any new code path that touches chat (including the test-message queue) must check this flag itself; don't assume `self._channels` membership implies the feature is "on" (there's up to one `SYNC_INTERVAL_SECONDS` of lag between toggling off and the channel actually being dropped from `self._channels`).
  - Beyond `checkin`/`reply`, there are **built-in chat commands** always available once chat is enabled (checked as a fallback after custom commands, so a custom command can shadow one if an admin is determined to): `포인트` (check balance), `도박` (start a chat-based gambling round using `points_gambling_config`/`points_gambling_options` — the same config the Discord `/포인트도박` slash command uses), `투표 <번호>` (vote, bet deducted immediately, one vote per round, no changing), `도박종료` (settle). `!도박`/`!도박종료` require the sender to be the streamer (see the `user_id == channel` heuristic above) or a manager per the *existing* `guild_config.mod_role_id`/`mod_managers` system (`utils.checks.member_is_mod_or_admin`) — there is deliberately no separate chzzk-specific manager role. Winner/payout math (`resolve_gambling_winner`/`calc_gambling_payout`) lives in `utils/gambling.py`, shared with `cogs/points.py`'s Discord-Poll-based gambling since the settlement formula is identical even though the two flows collect votes completely differently (poll gambling deducts bets at settlement time by scanning Discord's own Poll API; chat gambling deducts at vote time since the bot controls the vote command directly).
  - A guild can only ever have one *unsettled* `chzzk_gambling_sessions` row — enforced by a partial unique index (`idx_chzzk_gambling_sessions_one_active ... WHERE settled=0`) rather than application-level check-then-insert, since two near-simultaneous `!도박` triggers (e.g. real chat + the test-message queue) are two separate asyncio tasks and a SELECT-then-INSERT race is real.
  - **Web test-message queue** (`chzzk_chat_test_queue` table, dashboard "실시간 채팅 미리보기" panel, `POST /api/chzzk/{guild_id}/chat-test`): lets you exercise any chat command from the dashboard without a live Chzzk broadcast. The bot polls this table (`test_queue_loop`, `TEST_QUEUE_INTERVAL_SECONDS=2s`) independently of the real session connection — it only needs `chzzkpy` importable, not `CHZZK_CLIENT_ID`/`SECRET`, so it still works before a Chzzk OAuth app is registered locally. If the channel is already really subscribed it reuses the live `user_client` (so the response also posts to real chat); otherwise it uses `_NullChatClient` (a no-op stand-in) and only the dashboard's "in/out" log reflects it. Both this path and real chat funnel through the same `_process_message(entry, message)` so command dispatch never has two implementations to keep in sync.

- **OBS overlay widgets** (`chzzk_subscriptions.overlay_token`, public/unauthenticated endpoints under `GET /api/chzzk/overlay/{token}/...` in `chzzk_router.py`, pages under `web/frontend/app/overlay/{gambling,missions}/[token]`): a per-guild secret token (not a JWT) identifies the guild, because OBS Browser Sources cannot send custom headers. Two independent widgets share one token: `gambling-status` (live `!도박` round) and `missions-status` (active rows from the 포인트 미션 관리 tab, shown donation-goal style — there is no numeric goal/target concept in the `missions` schema, so it's a plain completion counter, not a progress bar). The public pages and the dashboard's own "오버레이" tab preview render through the *same* components (`GamblingOverlayView`/`MissionsOverlayView`) and the *same* polling hook (`web/frontend/lib/useOverlayPoll.ts`) — don't add a one-off fetch loop in either place, they will drift.

**Critical shared-state detail:** the Discord bot and `web/backend` are two separate Python processes that read/write the **same SQLite file** (`bot.db`, WAL mode via `aiosqlite`). In production (Railway) both run in one container via `start.sh` — bot in the background, backend in the foreground serving `$PORT`. Any schema or data change made by one is immediately visible to the other; don't assume isolated state between bot and backend code.

Slash commands sync **globally only** (`main.py`'s `setup_hook`) — there is deliberately no per-guild copy in normal operation (the owner-only `sync guild` prefix command exists only for fast local testing, and `clearguild` exists to purge accidental per-guild duplicates).

## Database

`database/db.py` owns schema for the entire project (both bot and backend import from here). There is no migration framework:
- Base schema is one `executescript` call in `init_db()`.
- Every schema change made *after* that baseline is appended as a new entry in the trailing list of raw SQL strings (`ALTER TABLE ...` / `CREATE TABLE IF NOT EXISTS ...`), each run individually and wrapped so failures (e.g. column already exists) are silently ignored.
- When adding a column or table, follow this same append-only pattern — don't rewrite the base `executescript` block, and don't introduce a new migration tool.

## Environment variables

No `.env.example` exists; required variables (see `.gitignore` — `.env` is never committed) are: `DISCORD_TOKEN`, `OWNER_ID`, `DATABASE_URL`, `DISCORD_CLIENT_ID`, `DISCORD_CLIENT_SECRET`, `DISCORD_REDIRECT_URI`, `JWT_SECRET`, `CHZZK_POLL_INTERVAL`, `FRONTEND_URL`, `CHZZK_CLIENT_ID`, `CHZZK_CLIENT_SECRET`, `CHZZK_REDIRECT_URI`. The last three are read by both `web/backend` (OAuth flows) and, as of the realtime chat feature, the bot process itself (`cogs/chzzk_chat.py`) — same values, both processes.

## Conventions

- Slash command names and cog user-facing strings are in Korean; keep new commands consistent with that.
- No test suite, linter, or CI currently exists in this repo — there's nothing to run before considering a change complete beyond manual verification.
- No branch/commit convention is enforced; this repo is committed to directly on `main`.
- Reuse the shared frontend primitives instead of re-copying their markup: `components/Switch.tsx` (toggle switches), `components/overlayPillStyle.ts` + `lib/useOverlayPoll.ts` (overlay widgets). These were extracted after the same toggle/polling code drifted slightly across 3 copies.
- When adding a new always-on chat command in `cogs/chzzk_chat.py` (like `포인트`/`도박`), add its trigger word to `_RESERVED_TRIGGERS` in `web/backend/routers/chzzk_router.py` so admins can't shadow it with a custom command.
